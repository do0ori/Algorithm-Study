# 문제 4. Read/Write 스케줄링 시뮬레이션

## 문제 설명

문자열 배열 `arr`에 대해 여러 프로세스가 `read` / `write` 작업을 요청합니다.

규칙은 다음과 같습니다.

1. 읽기(read)는 여러 프로세스가 동시에 수행할 수 있습니다.
2. 쓰기(write)는 한 번에 하나의 프로세스만 수행할 수 있으며, 쓰기 중에는 다른 읽기/쓰기는 모두 대기해야 합니다.
3. 읽기 작업이 진행 중일 때 쓰기 요청이 하나라도 대기열에 들어오면, 그 이후에 도착한 새로운 읽기 요청도 즉시 실행되지 못하고 대기해야 합니다.
4. 대기 중인 작업의 우선순위는 **write > read** 이며, write끼리는 요청 시간이 빠른 순서대로 처리합니다.

각 process 문자열 형식은 다음과 같습니다.

- `"read t1 t2 A B"`
- `"write t1 t2 A B C"`

의미:

- `t1`: 요청 시각
- `t2`: 작업 소요 시간
- `A..B`: 작업 구간 (1-indexed inclusive로 가정)
- `C`: write 시 덮어쓸 문자열

반환값은

1. 완료된 read 작업들의 읽은 문자열들을 실행 순서대로 담고,
2. 마지막에 전체 작업이 끝난 시간을 문자열로 변환해 맨 뒤에 추가한 벡터입니다.

## 제한사항

- `5 ≤ arr.size() ≤ 1000`
- 각 원소는 문자열
- `1 ≤ processes.size() ≤ 1000`
- `processes`는 `t1` 기준 오름차순 정렬
- 모든 `t1`은 서로 다름

## 복원 가정

원문이 없어 다음을 가정했습니다.

- `A, B`는 **1-indexed inclusive** 입니다.
- read는 **실제로 시작하는 시점의 배열 상태**를 읽습니다.
- 동시에 시작하는 여러 read의 결과는 **시작 순서(=요청 순서)** 로 결과에 담습니다.
- write의 반영은 구현상 **해당 write 작업이 시작될 때 즉시 배열에 적용**됩니다.

## 풀이 개요

writer-priority reader-writer lock을 이벤트 시뮬레이션으로 구현합니다.

- 현재 시각 이전에 도착한 작업들을 ready queue에 넣음
- active read가 있으면 그 종료 시각까지 진행
- write 대기 중이면 새 read는 막음
- 쓰기가 가능하면 가장 먼저 도착한 write 실행
- 아니면 현재 시점에서 시작 가능한 read들을 한 번에 묶어 동시 실행
